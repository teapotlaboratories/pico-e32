Flash-baked → self-loading

fake-08 now loads carts off the board’s onboard microSD. The new SD path is as separable from the shared fake-08 engine as the SD pins are from the LCD bus — a private four-wire bus, and a byte-identical source/.

Every pin number is quoted from board.cpp — the board owns both its LCD and (now) its SD wiring; the two buses share zero GPIOs. The disjoint hardware rhymes with the disjoint software seam — the mount is a reusable component as separable from the shared engine as the SD pins are from the LCD’s.

01 It was never actually parts-blocked

The port doc listed SD carts as parts-blocked. But the 3.5" board has an onboard microSD slot — only a card (a commodity) is needed, not a parts order. The "microSD bus shared with the LCD" note that implied a conflict was a carry-over from the 4" board (DP-7), wrong for rev-1. This report is the correction of record; the stale line in the board reference has now been fixed too.

02 Two buses, disjoint

The LCD is an i80 parallel bus (Bus_Parallel16) on 21 GPIOs; the SD rides its own SPI2_HOST on four. Their pin sets don’t intersect — 0 shared GPIOs — so the SD gets a fully private bus with no contention. Mount via esp_vfs_fat_sdspi_mount with format_if_mount_failed=false (never auto-wipe a card).

03 The cart ladder, graceful by contract

At boot: setCartDirectory(/sdcard) → listcarts() → if a cart is present, vm.LoadCart(carts[0], false) (the std::string overload — a VFS file read); else fall through to the flash cart. A mount failure logs a warning, frees the bus, and returns — it never aborts boot, so there’s always something to run.

The mount is a board-agnostic component (components/sdcard_spi) that never names a board; the board supplies its wiring through board_sd_config(), and an owns_bus flag lets a future board share the SPI bus with its display. The scan methods are in the fork’s platform binding, and the file read + cart-load path were already in shared source/ — so source/ stays byte-identical to upstream fake-08.

04 Two gotchas between “mounts” and “loads”

Case. FATFS readdir returns 8.3 short names UPPERCASE (HELLO.P8), and fake-08’s isCartFile checks the extension case-sensitively (it accepts .p8 / .png, incl. .p8.png) → carts were hidden. The fork’s listcarts lowercases a copy for the test but keeps the original-case name in the path (FAT open is itself case-insensitive).
Long filenames. Need CONFIG_FATFS_LFN_HEAP (+ MAX_LFN=255, UTF-8). IDF applies sdkconfig.defaults only when generating a fresh sdkconfig, so a pre-existing build silently ignores a newly-added option — it needs a fresh generation.

05 Verified on hardware

Observed on the bench per PR #10; the log strings are confirmed in source.

  • A real 32 GB SDHC card mounts (SD mounted at /sdcard).
  • Empty card → graceful fallback to the flash cart — no crash.
  • A seeded hello.p8 was loaded from SD and rendered ("LOADED FROM SD!" on the panel).
  • Clean production build boot: SD mounted → loading SD cart: /sdcard/HELLO.P8 → entering GameLoop.

06 Follow-ups

  • Cart selection (a chooser, not just the first cart) needs input — parts-blocked. Today: first cart by readdir order, deterministic, logged.
  • Line pull-ups are the #1 SD-SPI failure mode; a flaky socket may need them, or dropping max_freq_khz from 20 MHz (a config field the board sets in board_sd_config).
