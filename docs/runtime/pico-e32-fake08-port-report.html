TL;DR The short answer

Under firmware/, the whole fake-08 port is one small app: firmware/pico-e32-fake08/ (4 files, ~102 lines). Its job is just the ESP-IDF entry — app_main brings up the board, runs fake-08’s boot sequence, embeds a flash cart, and hands control to fake-08’s own game loop.

The actual porting work — the Host binding (ESP32Host.cpp) — deliberately does not live under firmware/. It lives in the vendored fork at components/fake08/platform/esp32/, following fake-08’s own per-platform convention. And fake-08’s source/ stays byte-identical to upstream.

01 Where the code lives

Two places. This repo (pico-e32) holds the app, the board driver, and the shared VM. The fork (teapotlaboratories/fake-08 @ pico-e32, pulled in as a submodule) holds the untouched fake-08 runtime plus the new ESP32 platform binding. Only the blue boxes are new code.

New = the ESP32 app + the platform binding + the component build. Kept/shared = fake-08’s runtime (byte-identical) and the z8lua VM. The port is a thin shell around an unchanged upstream.

02 What’s under firmware/

The direct answer. This is the entire app — a thin ESP-IDF project that wires the board and a cart into fake-08’s runtime, then calls fake-08’s own loop.

firmware/pico-e32-fake08/ ├─ CMakeLists.txt — IDF project; EXTRA_COMPONENT_DIRS → ../../components (11 ln) ├─ sdkconfig.defaults — 32 KB main stack for the Lua pcall path (10 ln) └─ main/ ├─ CMakeLists.txt — REQUIRES fake08 z8lua LovyanGFX driver esp_timer (7 ln) └─ main.cpp — app_main + boot sequence + flash .p8 cart the heart (74 ln)

Everything else the port needs — the Host implementation, the PICO-8 runtime, the VM — is pulled in as components (fake08, z8lua, LovyanGFX) and the board driver. So firmware/ stays tiny on purpose: it’s the system entry, not the port logic.

03 How a frame happens

main.cpp’s app_main mirrors fake-08’s desktop source/main.cpp boot order, then hands off to fake-08’s own Vm::GameLoop() — the frame loop is fake-08’s, not ours (that’s the 1-to-1 rule).

Only drawFrame is our code inside the loop — it unpacks fake-08’s 128×128 4-bpp framebuffer through an RGB565 palette LUT, 2× scales it to 256×256, and blits once. Everything else is fake-08’s or the board’s.
