drawFrame: 16.5 → 3.6 ms

The PICO-8 draw path was ~9× the VM and nobody had measured it. One buffer moved from PSRAM to internal SRAM. The frame is now VM-bound.

Measured on real Celeste (ESP32-S3), esp_timer around Step() vs drawFrame(). Step is the title screen (light); drawFrame is content-independent (always 128×128 px + one 256×256 blit). Ceilings are compute-bound (1000/(step+draw)), not paced or achieved fps.

01 The blind spot

Nobody had measured the real frame. Gate #2 (luabench) timed the z8lua VM with every draw call stubbed — so the actual Step() + drawFrame() cost was unknown, even though it's the load-bearing number for the 30-vs-60 fps policy. The draw-only port renders correctly; this is the first look at what it costs.

02 How it was measured

An opt-in build in firmware/pico-e32-fake08/main/main.cpp — under #ifdef MEASURE_FPS it swaps fake-08's vm->GameLoop() for an instrumented loop that times vm->Step() (the VM's _update/_draw) vs host->drawFrame() separately with esp_timer_get_time(), paces via waitForTargetFps(), and logs every 30 frames over UART:

f%ld: step=… draw=… total=… ms | max %.1f fps | worst …

Enable with DEFS='-D CELESTE=1 -D MEASURE_FPS=1'; read the UART with raw stty 115200 + cat (idf.py monitor won't run headless). Target is the real Celeste cart — it parks on the title (input stubbed), so the step number is the title screen; drawFrame is representative regardless.

03 The finding — drawFrame was PSRAM-bound

drawFrame was 16.5 ms — about 8.7× the VM. The boot log named the culprit:

W · scaled fb fell back to PSRAM fb = 0x3c1004f8

That address is a runtime observation, and it's consistent with the ESP32-S3 map: PSRAM is 0x3C000000+, internal DRAM 0x3FC88000+. The 256×256 scaled frame is 128 KB — too big for internal DMA-capable SRAM — so it fell back to PSRAM, and both the per-pixel writes and the full-frame blit-read then paid PSRAM latency.

04 ★ Where the frame lived

The whole story is the region: the buffer landed in the 0x3C… PSRAM range; the fix puts it in the 0x3FC… internal-SRAM range. Addresses are from the boot log, consistent with the S3 memory map.

05 The fix — strips from internal SRAM

Don't materialise the 128 KB framebuffer at all. Build and blit the frame in 8 strips of 32 panel rows through a single 16 KB internal DMA buffer (MALLOC_CAP_INTERNAL | MALLOC_CAP_DMA; strip = 0x3fcc99c8 — internal, no fallback). Every write and the blit DMA now stay in fast SRAM. The 4-bpp nibble-unpack is also inlined — one byte src[x>>1] yields two pixels (low b&0x0f, high b>>4) — dropping the per-pixel getPixelNibble call.

In the fork's ESP32Host::drawFrame @ 8f4b6a6 — source/ byte-untouched, one file, +37 / −25.

06 What it means — draw-bound → VM-bound

drawFrame is now a fixed 3.6 ms. Against Gate #2's measured gameplay VM (~15.8 ms average, 5–40 ms per room), the frame budget flips: the display is settled and the interpreter is the constraint. Bars below: ■ VM Step + ■ drawFrame, against the 33.3 ms / 30 fps budget (amber line).

So the draw path is no longer the question. The remaining frame-rate lever is the z8lua interpreter on heavy rooms — the open speedup item, to be tackled only after profiling a representative cart on the S3.

07 Follow-ups

  • A direct gameplay Step measurement (drive Celeste past the title via scanInput) would firm up the ~15.8 ms extrapolation — currently the step number is the title screen.
  • drawFrame could inline further / use 32-bit stores, but at 3.6 ms it's no longer the bottleneck.
