The real bottleneck — not the display

The core insight: interface class > screen size

The one decision that shapes everything is how the panel is wired, not how big it is. A GRAM-backed i80/SPI panel takes bounded, on-demand pushes. An RGB panel has no frame memory, so the ESP must stream the whole framebuffer out of PSRAM continuously — and the S3 has no hardware arbitration to protect that stream.

CPU + scanout share the PSRAM port → permanent screen drift, reproduced on this exact panel. Perverse: the heavy carts that need the 8 MB are the ones that drift it.

Where the bytes live (ILI9488 i80 path)

Because the panel holds the image in its own GRAM, the ESP keeps only small buffers in fast SRAM and hands the entire PSRAM to the interpreter. This is why 2 MB is enough to start — and for the large majority of carts.

The one limit

Only a cart pushing near PICO-8's full 2 MB Lua ceiling runs short of GC headroom on a 2 MB part — a minority of the library, and a later concern already covered by the ordered 8 MB (4″) board. No module to buy to begin.

Phase 0 — the two project-killers

Everything hinges on two questions, both answerable on the ILI9488 for $0. Run them before buying parts or designing anything.

🅐  Display path esp_lcd i80 → ILI9488

A1 — Flash an i80 example, get colour bars on the panel (ESP-EYE-verified).
A2 — Blit a 128×128 indexed image → palette-expand to RGB565 → 2× scale to 256², letterboxed → i80 DMA; time the flush loop.
🚦 Gate #1 — ≥ 30 fps sustained. (Bus theoretical ≈ 130 fps; expected to pass easily.)

🅑  Lua throughput the actual risk

B1 — Build z8lua (fix32, Lua 5.2) as an ESP-IDF component, compiled as C++ (fix32 is a C++ type) with -DLUA_USE_LONGJMP.
B2 — Micro-benchmark: fixed-point loop, sin() sweep, table churn — heap in SRAM vs PSRAM, timed with esp_timer.
🚦 Gate #2 — Lua within ~2–3× of PICO-8's ~4.2 M inst/s budget (≤ 33 ms/frame). PicoPico ran Celeste at ~9 ms/frame on a slower LX6 — a safe lower bound.
Then Gate #3 — a trivial cart

A minimal ESP32Host (draw + timing only) runs a hand-written moving-sprite .p8 at ≥ 30 fps on the panel. That closes the loop end-to-end on the safe i80 board.

The runtime — port fake-08, write one host layer

fake-08 (MIT) is a clean-room PICO-8 runtime with the highest open API coverage. Its VM is portable and bare-metal-clean; all OS coupling sits behind a single Host class. You write one ESP32Host — the VM is reused as-is.

Verified low-risk on the VM axis

Heap grows dynamically (no static 2 MB reservation); interpreter needs only malloc + setjmp/longjmp; fix32 keeps all cart math off the FPU. Place the Lua heap in PSRAM via a heap_caps allocator; keep hot state in SRAM. Split cores: Lua on core 0, display + audio on core 1.

The open risk (Gate #2)

No one has run fake-08 on an ESP32 yet — the proven ESP32 PICO-8 runtimes are tac08 and PicoPico. fake-08 is the better base; Phase 0 converts "should work" into "does work."

Gate #5 & the fallback ladder

Gate #5 — the 4″ RGB drift soak test (hardware, not arithmetic)

Promote the 4″ board to target only if all three hold: (1) sustained ≥ 30 fps for minutes with bounce buffers + RGB_RESTART_IN_VSYNC + XIP-from-PSRAM, at a PCLK above the ~40 Hz IPS flicker floor; (2) a cart with a ~1.5–2 MB heap spilled to PSRAM still holds the refresh deadline; (3) a live SD cart-load causes no drift. If it fails → ship the 4″ for light/moderate carts, or go custom (N16R8 module + a GRAM-backed i80/SPI panel).

If even the LX7 is too slow (Gate #2 fails badly)

Fallback ladder: 30 fps baseline → restrict the cart list → bytecode precompile + SRAM-arena allocator → ESP32-P4 (≈2× single-thread Lua headroom) as the nuclear option. Honest non-MCU floor: a Raspberry Pi Zero 2 W running the official PICO-8 binary.
