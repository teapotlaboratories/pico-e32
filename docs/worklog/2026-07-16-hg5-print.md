## The font: authored, not sourced

The [host-graphics doc](../runtime/pico-e32-host-graphics.md#font-licensing--take-it-from-the-source-not-from-an-emulator-hg-5)
worked out that PICO-8's font is **CC-0** and that an emulator's *file* (e.g. fake-08's `fontdata.cpp`,
MIT, with no font copyright notice) shouldn't be copied — the data may be transcribed, but not the file.

For HG-5 I went one step more conservative: **the glyphs are authored** as 3×5 row-strings
(`" # "`, `"# #"`, …), in `assets/pico8_font.h` (the firmware includes it and decodes to bitmaps at init).
This carries **no third-party data at all** — CC-0 or otherwise — so none of the provenance question
applies. The trade-off is fidelity: these are *PICO-8-style* glyphs, not pixel-identical to PICO-8's.

**Backlog RT-FONT — rebuild `pico8_font.h` from a real PICO-8 font.** The header is a hand-authored
*placeholder*; PICO-8's actual font is CC-0 (8 bytes/char, 8×8 bitfield, **bit LSB = leftmost pixel** — from
the Lexaloffle BBS + manual). Export it (a PNG, or a font-memory dump) and convert it — ideally via an
`assets/gen_font.py` that emits `pico8_font.h`, the same pattern `gen_celeste_cart.py` uses for the sprite
sheet — for pixel-exact glyphs with zero hand-drawing. Blocked only on obtaining the font bytes (no PICO-8
on this dev box). A separate `.h` isolates the placeholder so regeneration is a one-file swap that never
touches the renderer. Tracked in [`docs/runtime/pico-e32-host-graphics.md`](../runtime/pico-e32-host-graphics.md).

Geometry matches PICO-8 so layouts line up: **3px-wide glyphs, cursor advances 4px per char, 6px per
line.** PICO-8 renders lowercase as its caps glyphs, and Celeste's strings are lowercase — so `a-z` map to
the same glyphs as `A-Z` (`memcpy(g_font[c], g_font[c-32], 5)`).

## `print()`

`print(str, [x], [y], [col])` — draws each char from the font, `\n` starts a new line, and it leaves the
cursor at the next line (PICO-8 style) so a bare `print(str)` continues below. `x`/`y` default to the
cursor; `col` to the current colour. Null-safe on everything, so the trivial cart is unaffected.

## Verification — the glyph set + Celeste's real strings

`P8_TEXTTEST=1 make dump` renders the whole printable set and Celeste's actual UI strings:

```
abcdefghijklm  nopqrstuvwxyz          <- lowercase, shown as caps
0123456789     .,:;!?-+=/%*#()
old site       deaths:12
x1000          100 m
matt thorson   noel berry
summit         time 3:07
```

All **legible** in the dumped frame — the alphabet, digits, punctuation, and every Celeste UI string
(`OLD SITE`, `DEATHS:12`, `X1000`, `MATT THORSON`, `NOEL BERRY`, `SUMMIT`, `TIME 3:07`) read cleanly. The
weakest glyph is `N` (a 3px diagonal), still readable in `THORSON`/`NOEL`.

**Trivial-cart `fb_hash` unchanged** (`5e851a75`) — `print` is null-safe, so a cart that never calls it is
untouched. No Gate #3 regression.
