# 2026-07-16 — HG-3: `spr()`, the first real cart drawing

**Goal:** implement PICO-8's `spr()` in the host draw API — the core of every cart, and the gap between
"Celeste's logic runs" (Gate #2) and "Celeste draws a pixel". Verified **camera-independently** via the
host frame-dump → PNG harness (HG-2), against Celeste's real sprite sheet (HG-1).

**Status:** ✅ **done and verified.** `spr(n,x,y,[w,h],[flip_x,flip_y])` + a minimal `palt()`. Built on
the [#6 branch](2026-07-16-esp-lcd-vs-lovyangfx.md); **commit held for after 17:00 Pacific** (work-hours
rule) — the code and evidence are in the tree.

---

## What landed

`firmware/pico-e32-host/main/host_main.cpp`:

- **`spr(n,x,y,[w,h],[flip_x,flip_y])`** — blits sprite `n` (an 8×8 tile, or a `w*8 × h*8` block) from
  the active sheet to `(x,y)`, skipping transparent colours. Sprite `n`'s top-left in the 128×128 sheet
  is `(n%16*8, n/16*8)`; `flip_x`/`flip_y` mirror the whole block about its own centre.
- **`g_gfx`** — the active sprite sheet pointer (128×128 palette indices, the unpacked layout HG-1
  emits as `CELESTE_GFX`). **Null-safe**: the trivial cart sets no sheet, so `spr()` is a no-op there and
  the Gate #3 output is untouched (verified: `fb_hash` unchanged, below).
- **Minimal `palt()`** — default colour-0 transparency, with `palt()`/`palt(c,t)` to change it. The
  colour-*remap* half of `pal`/`palt` stays HG-6; this is only the transparency table `spr()` needs to
  look right. `spr`/`palt` moved out of the no-op stub list.

## Verification — three mechanics, one camera-free harness

Added a host-only spr test behind `P8_SPRTEST=1` (so the default `make dump` still runs the trivial
cart). It prefers Celeste's **real** `CELESTE_GFX` (via a guarded `__has_include("celeste_cart.h")`, so
the build still works without the gitignored asset — falls back to a synthetic sheet) and runs a cart
that exercises all of `spr()`:

1. **A 64-sprite grid** — `for ry,rx: spr(ry*16+rx, rx*8, ry*8)` — reproduces the sheet's top 32 rows via
   individual `spr()` calls. Renders as recognisable Celeste sprites: Madeline's faces, blue balloons,
   the chest, strawberries, springs, clouds, grass, flowers, spikes. Proves sprite decode + position +
   that the dark-blue `cls(1)` shows through the transparent (colour-0) pixels.
2. **A 2×2 (`w=h=2`) block** — `spr(0,80,104,2,2)` — assembles sprites 0/1/16/17 into a **full Madeline**
   (head + body). Proves the `w`/`h` multi-tile path.
3. **An asymmetric "F", flipped four ways** — proves `flip_x`/`flip_y`.

### The gotcha: Celeste's sprite 0 is blank

The flip test first used sprite 0 — and passed *vacuously*: sprite 0 is **blank** in Celeste (Madeline
starts at sprite 1), so "flip of nothing == mirror of nothing" is trivially true and proves nothing. Fix:
paint a known **asymmetric, non-blank "F"** into sprite 64 (at sheet `(0,32)`, below the grid so it
doesn't disturb it) and flip-test *that*. The check is now meaningful — and it's the L-pattern lesson
again: an asymmetric known pattern beats "looks roughly right."

### Result — exact mirrors, not approximate

Read straight from the dumped framebuffer (`frame_000.raw`):

```
F non-bg pixels: 17            (non-blank ✓)
F asymmetric under X: True     (so a mirror is actually different — the test isn't vacuous)
F asymmetric under Y: True
flip_x  == horizontal mirror of normal:  True
flip_y  == vertical   mirror of normal:  True
flip_xy == both-axis   mirror of normal: True
```

And visibly, the four F's point four different ways (normal / flip-X / flip-Y / both). Numeric + visual
agree.

### No regression

The trivial cart (Gate #3 / device path) prints `fb_hash=5e851a75` — **byte-identical** to before HG-3.
`spr()` being null-safe means registering it changed nothing for a cart that never calls it.

## How to reproduce

```sh
cd firmware/pico-e32-host/host
make dump                       # trivial cart, fb_hash=5e851a75 (regression check)
P8_SPRTEST=1 P8_DUMP=frames ./host
../../../tools/p8_png.py frames/frame_000.raw frames/sprtest.png
# grid of Celeste sprites + 2x2 Madeline + four F's; the flip check is the numpy snippet above
```

(`assets/celeste_cart.h` must be generated first — `python3 assets/gen_celeste_cart.py` — for the real
sheet; without it the test uses the synthetic fallback.)

## Not covered here / next

- **`pal()` colour remap, `camera`, `clip`** — HG-6. `spr()` honours the transparency table but not a
  colour remap yet.
- **The panel.** This is a host-framebuffer proof (sprite decode/position/flip/transparency). That the
  *glass* shows it is a separate camera check (BC-2/BC-3) — but the existing host↔device `fb_hash` compare
  ties them: if a real cart's hashes match and the host image is right, the device framebuffer is right.
- **HG-4 — `map()`** is next, and it carries the shared-memory gotcha: `__map__` alone is only rooms 0–15
  of 32; the generator must also emit sheet rows 64–127 as map rows 32–63 (and `CELESTE_MAP[4096]` is
  used by luabench — resize with care). See the *shared memory* finding in
  [`docs/runtime/pico-e32-host-graphics.md`](../runtime/pico-e32-host-graphics.md).
