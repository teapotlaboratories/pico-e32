Celeste plays, but "fps is horrible"

2026-07-18 · the 60 Hz-resume fix · fake-08 port

Real Celeste boots end-to-end through fake-08's own GameLoop — then the owner played it and said the fps was horrible. Measuring showed the work at ~6 ms/frame (162 fps capable), so it was never the S3. Two pacing bugs: a 100 Hz FreeRTOS tick quantized the frame limiter (judder), and the host paced fake-08's 60 Hz-resume coroutine at 30 Hz — so 30 fps carts ran at half speed. Fixed with a 1 ms tick + setTargetFps(60). Verified 60 Hz resume / correct 30 fps motion, driven over serial. Plus an opt-in on-screen FPS HUD.

Measure first — the hardware is not the bottleneck

The app's MEASURE_FPS loop times Step() vs drawFrame() in isolation (no pacing). On Celeste:

stagecostnote
Step() — VM _update+_draw2.56 msz8lua with jump tables
drawFrame() — unpack → 2× → strip-blit3.61 msinternal-SRAM strips
total work~6.2 ms→ 162 fps capable

So "slow" was pacing, in two independent bugs.

Bug 1 — pacing quantization (the judder)

The 30 fps limiter (Host::waitForTargetFps) sleeps with vTaskDelay, which rounds to whole FreeRTOS ticks. The tick was the IDF default 100 Hz = 10 ms, so a 33.3 ms target quantized to ±10 ms — frames landed at 20/30/40 ms and juddered though the average was ~30 fps.

Fix: CONFIG_FREERTOS_HZ=1000 (1 ms pacing).

Bug 2 — half-speed (the real "slow")

After Bug 1 the delivered rate was a rock-steady 30.0 fps — yet still slow. fake-08's game loop is a coroutine designed to be resumed at 60 Hz (__z8_run_cart glue). It flip()s (yields) after each _draw, and for a 30 fps cart it burns an extra yield():

while true do
  if _update60 then _update_buttons(); _update60()
  else              yield();  _update_buttons(); _update()   -- extra yield = 30 fps
  end
  if _draw then _draw(); flip()  -- flip() yields (presents the frame)
  else               yield() end
end

The host clock (resume rate) and the cart's logical rate are different things. The host always resumes at 60; the coroutine self-divides. Our ESP32Host paced at 30 Hz (setTargetFps(30)) — upstream fake-08 uses 60 — so Celeste got 15 fps of motion.

Fix: host->setTargetFps(60) in the app (+ the ESP32Host default). This covers every PICO-8 cart — the platform defines only 30 (_update) and 60 (_update60); there is no other rate.

On-screen FPS HUD (-D SHOW_FPS=1)

Opt-in: the loop rate painted in the panel's right letterbox (x ≥ 288). The centred 256 px game blit never touches that strip, so the readout persists without a per-frame redraw and never covers gameplay. Reads 60 when healthy; drops if a room exceeds the 16.6 ms budget — a live perf meter. board_lcd_draw_fps() is the board's (LovyanGFX); ESP32Host calls it on change, gated in components/fake08/CMakeLists.txt.

Lesson

A steady average fps can still be half-speed. fake-08's loop is a 60 Hz-resume coroutine that self-divides 30 fps carts internally — the host must resume at 60, never at the cart's logical rate. Measure the loop resume rate and the game motion separately; they are not the same number.
