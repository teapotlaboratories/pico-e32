The shared-memory problem, resolved

PICO-8's map is 128×64 cells, but the __map__ section carries only rows 0–31. Rows 32–63 share memory with the bottom half of the sprite sheet (gfx rows 64–127), and Celeste uses it — rooms 16–31 live there. A map() fed only from __map__ draws half the game.

The generator packs gfx rows 64–127 into the map's second half. The two halves' tile statistics are near-identical — the first sign it's real level data.

Gate 2 (luabench) is unaffected — the consumer check held

CELESTE_MAP grew 4096 → 8192, and it's shared with pico-e32-luabench. Safe, because luabench's mget/mset clamp to y<32 (celeste_bench.cpp:17,22) — they only index [0,4096), and map=nop there (headless). The first 4096 bytes are byte-identical (fingerprint 408a1f31, max tile 104) and luabench builds clean against the 8192 array. Gate 2 sees exactly what it saw before.

The layer filter — a verified strict subset

Rendered room 1 with each single-bit layer and compared to the full render:

full room 1: 10827 lit px
  layer 0x01:  6720 px   subset-of-full = True
  layer 0x02:  6869 px   subset-of-full = True
  layer 0x04:  3916 px   subset-of-full = True
  layer 0x08..0x80:  0 px   (Celeste doesn't use those flag bits)
  union of all layers: 10785 px  (vs full 10827)

Every layered render is a strict subset; flag bits 0–2 carry content (real CELESTE_GFF via fget); the union reconstructs the full render minus 42 px — the flagless tiles, which PICO-8 draws only at layer=0. Correct behaviour, confirmed.
