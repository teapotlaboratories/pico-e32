Phase-0 Gate #1: ILI9488 i80 scaled-blit FPS

✓ FPS criterion passed (288 fps) ⚠ visual unverified
The 256×256 scaled blit runs at 288 fps on the ILI9488 (16-bit i80) — ~10× the ≥30 fps gate, and physically credible (~94% of the 20 MHz×16-bit bus). But the bench camera is unavailable, so I measured throughput, not that the correct pixels are on the glass.

Is 288 fps real? (yes)

288 fps sits right against the bus ceiling → a genuine transfer rate.

256×256×2 B × 288 = 37.7 MB/s vs the 40 MB/s bus → ~94% utilization. Init ran with no ESP_ERROR_CHECK abort (the app reached the loop), so the bus + panel IO are configured right. The real workload (128×128 → 2× letterboxed at 30 fps) needs only ~10% of this — huge margin.

⚠ Blocker: visual correctness not verified

Per .ai/AGENTS.md, display changes are confirmed by the bench camera — and it's unavailable: no /dev/video*, no capture tooling, no docs/hardware/ capture doc; the plan's intended cam is an ESP-EYE not set up here. The driver is explicitly UNVERIFIED, so a broken init could still clock DMA at 288 fps into a blank screen.

  • Not confirmed: colours, RGB/BGR order (SWAP_COLOR_BYTES), MADCTL orientation, panel not blank/garbled.

Follow-up: driver extracted to a reusable component

The inline ILI9488 driver is now components/ili9488 (ili9488_init/blit/fill/rgb565, board specifics via a config struct) so Gate #3 and the future ESP32Host share one driver. Rebuilt + reflashed → identical 288.0 fps, so the refactor is behavior-preserving (verified without the camera).
