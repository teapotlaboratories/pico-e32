z8lua: vendored copy → git submodule

✓ done  components/z8lua is a submodule tracking branch pico-e32 of teapotlaboratories/z8lua at d10747d. The vendored tree is byte-identical to upstream except one fix32.h patch; all ESP-IDF integration lives in the build files. Host build verified.

The core problem: a C++ dialect shipped as .c

z8lua is PICO-8's fixed-point Lua 5.2 dialect — LUA_NUMBER = z8::fix32, a C++ type — so the Lua core must compile as C++. The whole question was how to make that happen while touching upstream as little as possible.

Two tries

✗ First attempt (wrong). Rename all 30 compiled .c files to .cpp, and delete the units we don't compile. It builds, but it rewrites the entire vendored tree, turns every upstream rebase into a rename war, and buries the real divergence. "Nobody renames a .c to .cpp just to integrate."

✓ Correct approach. Leave the tree byte-identical to upstream; tell the build to use the C++ compiler — which is exactly what upstream's own makefile does (CC = g++).

Same source, two integrations. The build-layer one keeps the file byte-identical to upstream.

How the build does it

  • ESP-IDF (CMakeLists.txt): register the core .c files and set_source_files_properties(… PROPERTIES LANGUAGE CXX) so CMake invokes the C++ compiler; LUA_USE_LONGJMP keeps setjmp/longjmp error handling (C++ exceptions off).
  • Host (Makefile): compile the same .c with g++ -x c++.
  • Excluded units liolib/loslib/loadlib/ltests (host OS / dlopen deps linit never opens) are left out of the source list, not deleted.
