z8lua: vendored copy → git submodule

✓ done  components/z8lua is now a submodule tracking branch pico-e32 of teapotlaboratories/z8lua (a fork of jtothebell/z8lua) at 346ddf0. Host build verified; parent-repo wiring ships as a PR.

Why it needed a fork, not a plain submodule

z8lua is PICO-8's fixed-point Lua 5.2 dialect (LUA_NUMBER = z8::fix32). The vendored copy carried two deliberate patches without which it will not build for the ESP32-S3 — so a submodule pointing straight at upstream would silently drop them and break the build. The patches have to live in a fork we control.

The two divergences

  • .c → .cpp rename. fix32 is a C++ type, so every compiled translation unit must be C++.
  • fix32.h ctor patch. On xtensa-esp-elf int32_t is long, leaving plain int uncovered → int → fix32 was ambiguous under -std=gnu++2b. Added an implicit int / unsigned int constructor (no-op on x86, where int == int32_t).

The transformation

Upstream 4cf7de4 + 3 patch commits = branch pico-e32; the parent repo references its tip as a submodule gitlink.

Findings — checked, not from memory

  • All 30 .cpp files are byte-identical to upstream .c → the rename is pure, no smuggled edits.
  • Only fix32.h carries a content edit — exactly the documented hunk.
  • bench.h did not belong to z8lua. It was a stale duplicate of the canonical firmware/pico-e32-luabench/bench/bench.h (z8lua copy declared only run_bench; the real one also declares run_cases). The host Makefile's -I order made the build accidentally pick the stale copy — it compiled only because host_main.cpp calls just run_bench. Dropping it makes both builds resolve to the canonical header.

Follow-up — keep, don't delete

Commit 101a0c4 first deleted the units we don't compile (liolib/loslib/loadlib/ltests) and upstream's build files. Deleting vendor code just to keep it out of the build is more destructive than needed: both build systems compile only *.cpp (file(GLOB … "*.cpp") / $(wildcard …/*.cpp)), so leaving those units at their upstream .c extension already excludes them — no #define guard required.

  • Fork commit 346ddf0 restores all 8 files (a forward commit, so merged 101a0c4 stays reachable) and documents the convention: .cpp = compiled here; .c = upstream unit intentionally left uncompiled.
  • Verified: the compiled set is unchanged; benchmark still runs (~298 ms).
  • New rule in .ai/AGENTS.md: least-destructive vendor edits — exclude at the build layer or guard behind a compile flag; deletion is a last resort.

Verification

  • Tree parity: fork branch ≡ old vendored components/z8lua/, the only intended difference being the dropped stray bench.h. Upstream's .gitignore is retained (build-invisible — the CMake glob is *.cpp).
  • Compiles + runs: luabench host benchmark built through its real Makefile against the submodule — clean build, bench.h resolves to ../bench/bench.h, benchmark ran (fixed-point ~176 ms, sin/cos ~38 ms, table churn ~74 ms).
  • Caveat: the host is x86, so it can't exercise the xtensa fix32 int fix. That belongs to the on-device Gate #2 run.
