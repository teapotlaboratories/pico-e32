fake-08 port: vendoring + the ESP32 platform

2026-07-16 (UTC 07-17) · draw-only milestone

milestone reached The runtime is now a port of fake-08. Forked to teapotlaboratories/fake-08 @ pico-e32, added a platform/esp32 Host binding + an IDF component (source/ byte-identical), and a pico-e32-fake08 app. It boots on the ESP32-S3 and renders a flash .p8 cart upright on the panel through fake-08's own Vm::GameLoop and the shared z8lua VM — retiring the plan's one real unknown: nobody had run fake-08 on an ESP32.

The shape of the port

fake-08's Host is one concrete class; each platform links its own method definitions. We write one ESP32Host and reuse everything else — the Lua binding, the whole draw API, the cart loader, the frame loop. The board display driver we already had becomes drawFrame's back end.

Vendoring — the fork convention

The repo already forks upstream into the org and pins a project branch (that's how teapotlaboratories/z8lua @ pico-e32 works). fake-08 itself carries per-system port branches off master (wiiu-standalone, rg35xx-libretro, …), so an ESP32 branch fits its own convention. We forked to teapotlaboratories/fake-08, branched pico-e32, and added it as a submodule at components/fake08 — with fake-08's nested libs/z8lua left un-inited (we link the shared components/z8lua instead).

Build-integration fixes — meeting a modern toolchain

fake-08's per-platform GNU Makefiles meet GCC 14.2 / C++23-default ESP-IDF. Three adaptations, all in the fork's CMakeLists.txt — none touch source/:

SymptomFixWhy
lodepng::decode(…, filename) — no matching overload Leave lodepng disk I/O enabled cart.cpp:310 references the file overload unconditionally; fopen is newlib, the path is dead code here
lerp conflicts with a previous declaration Compile the component at -std=gnu++17 fake-08 is C++17; its global lerp collides with C++20 std::lerp
fake-08 needs its VM's lua_setpico8memory / fix32 REQUIRES z8lua — the shared fork Our z8lua exports exactly that API (lua.h:260); the whole runtime compiled against it with no drift

The PSRAM Lua-heap hook the plan called for (vm.cpp:300) turned out unnecessary for boot: CONFIG_SPIRAM_USE_MALLOC already backs large allocations with PSRAM. Keeping the hot state in internal SRAM is a later optimization. Result: an 846 KB image, 19% partition free.

On the panel — and one twist

First flash: fake-08 booted and rendered — but rotated 90° (vertical bars became horizontal, text went sideways). Was it the port, or had the bench camera moved? Reflashing the trivial L-pattern host and capturing in the same position gave an exact match to the archived reference — camera unchanged. So the display path itself renders PICO-8 content rotated 90°. The L-pattern never revealed it (its top+left bars render as a plausible on-panel "L"); fake-08's text is the first content that made it obvious. HG and fake-08 share the identical blit, so it's a whole-path property — worked around in drawFrame (pre-rotate 90° CW) and logged for a systemic board.cpp fix (DP-8). After the fix: bars vertical, text left-to-right, a live frame counter. Milestone clean.
