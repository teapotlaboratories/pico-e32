SD cart loader

2026-07-17 · fake-08 loads carts from the onboard microSD

The fake-08 port goes from a one-cart flash demo to a self-loading runtime: it mounts the board's onboard microSD (SPI2), scans for .p8/.p8.png, and loads one — falling back to the flash cart when there's no card. Verified on a real 32 GB SDHC card: mount, graceful empty-card fallback, and a seeded cart rendering "LOADED FROM SD!" on the panel.

Not actually parts-blocked

The 3.5" board has an onboard microSD slot (SPI: CS=1/MOSI=2/MISO=41/CLK=42). The port doc called SD "parts-blocked", but only a card — a commodity — is needed. The SD pins are disjoint from the i80 LCD, so the SD gets a private SPI2 bus, no contention (the "shared bus" note was a 4"-board carry-over, DP-7 — confirmed against board.cpp).

The SD rides its own SPI2 bus, independent of the i80 LCD. At boot the app tries an SD cart first and falls back to the flash cart — mount failure never aborts.

What changed (source/ byte-identical)

  • app — new sd_mount.{h,cpp} (esp_vfs_fat_sdspi_mount, graceful); main.cpp sets /sdcard via the shared setCartDirectory() and runs the ladder (LoadCart(carts[0], false) → VFS file read).
  • fork ESP32Host — un-stub listcarts() (scan the dir, ported from gcw0) + getCartDirectory().
  • shared source/ untouched — the file read (get_file_contents), setCartDirectory, and the absolute-path cart load (cart.cpp) already exist. REQUIRES gains fatfs sdmmc esp_driver_sdspi esp_driver_spi.

Two gotchas fixed

Case. FATFS readdir returns 8.3 short names UPPERCASE (HELLO.P8), and fake-08's isCartFile checks .p8 case-sensitively → carts hidden. The fork's listcarts lowercases a copy before the check; FAT open is case-insensitive, so the path keeps original case.
Long filenames. Need CONFIG_FATFS_LFN_HEAP — and IDF applies sdkconfig.defaults only when generating a fresh sdkconfig, so an existing build silently ignored it. After a regen, long/lower-case names survive.

Follow-ups

  • Cart selection (a chooser, not just the first cart) needs input — parts-blocked. Today: first cart by readdir order, deterministic, logged.
  • Line pull-ups are the #1 SD-SPI failure; this card worked, but a flaky socket may need them or a lower max_freq_khz (a commented one-liner in sd_mount.cpp).
