ESP32Host graphics: cart assets, the harness, and the font question

✓ HG-1 done ✓ HG-5 unblocked ⚠ nothing renders yet
Started the PICO-8 graphics surface — the one Phase-1 item that is neither parts- nor camera-blocked. The sprite sheet is extracted and verified by eye; the font turns out to be CC-0 from Lexaloffle, so no emulator's copy is needed. Two findings fell out that change other people's work — one for HG-4, one a caveat on Gate #2's headline number.

This log is updated as the work happens, at each checkpoint — unlike the bench-camera log, which was reconstructed at the end.

Why this, now

Celeste's logic already runs at frame rate (Gate #2, ~15.8 ms/frame) and the display path is proven (Gate #3, 161.5 fps) — but spr/map/print are no-op stubs, so no Celeste pixel has ever been drawn. Input, audio and SD are all parts-blocked; graphics is not. And it is verifiable without the bench camera, which matters while BC-1 blocks the visual gates.

HG-1 ✅ — __gfx__ extraction

gen_celeste_cart.py already pulled __lua__, __gff__ and __map__ out of the cart, but never __gfx__ — which is precisely why spr was a stub: there were no sprites to draw.

The section is 128 lines × 128 hex chars, one hex char per pixel. Note this is not PICO-8's in-memory packing (2 px per byte, low nibble = left), so it decodes as a straight char→value read — no nibble swap. Emitted as CELESTE_GFX[16384], unpacked (1 index per byte) rather than 8 KB @ 4 bpp: it keeps the spr() inner loop free of a shift/mask per pixel, and 16 KB of flash is noise against a 78%-free partition.

Verified by looking at it, not by asserting a byte count: the decoded sheet rendered to PNG shows Madeline's sprites, balloons, snow, spikes, the chest, strawberries and the CELESTE title logo, with all 16 palette entries in use. Round-tripped out of the generated header, so it tests the real artifact.

HG-2 🔨 — the verification harness

tools/p8_png.py renders any indexed buffer (one palette index per byte) to PNG through the PICO-8 palette. Stdlib zlib only — no Pillow, so it runs anywhere the repo does. Its PAL must stay identical to PAL888 in host_main.cpp; that equality is the point, since the tool exists to show what the firmware would draw.

Still to do: the host-side dump of live frames — -DHOST_MAIN already runs the cart and prints fb_hash; it needs to also write the framebuffer out.

Finding 1 — the sheet and the map share memory (changes HG-4)

The rendered sheet's bottom half looked like noise. It is not a decode bug: PICO-8 aliases sprite rows 64–127 onto map rows 32–63 (0x1000–0x1FFF), and Celeste stores map data there. Decoding that region as map bytes (tile = px[2x] | (px[2x+1] << 4)) yields exactly 4096 bytes whose statistics match the known-real __map__:

zeros (empty)tile id < 64max iddistinct
gfx rows 64–127, decoded as map40%95%12792
__map__ rows 0–31 (known real)36%96%104—

That is map data, conclusively. Celeste's map is 128×64 cells — 32 rooms of 16×16 in an 8×4 grid — and __map__ carries only half of it. From the cart: level_index() = room.x%8 + room.y*8 and load_room reads mget(room.x*16+tx, room.y*16+ty) (celeste.p8:63, :1118, :1132).

Half the world lives in memory the sprite sheet shares. Feeding map() from __map__ alone renders rooms 0–15 and silently empties 16–31.

Finding 2 — a caveat on Gate #2's headline number (not a bug)

celeste_bench.cpp:17 bounds-checks mget at y<32, returning 0 above — the exact half-map behaviour. This does not invalidate the ~15.8 ms/frame result: the bench deliberately averages levels 3–15 only (for (int lv=3; lv<=15; lv++)), all of which live in rows 0–31, and the code says so (:119 — “all levels 0-15 are in __map__”). The measurement is honest and correctly scoped.

But: levels 16–30 were never benchmarked — roughly half the game, and the later, generally busier half — because their map data was not extracted. Gate #2's “solid 30 fps” conclusion therefore rests on the first half. Now that HG-1 decodes the shared region, the bench can cover all 31 gameplay levels; if the later rooms are denser the average moves, and that feeds straight into the 30-vs-60 fps policy Gate #4 must set. (Level 31 is the title screen.)

HG-5 ✅ unblocked — the font is CC-0 from the vendor

print needs the PICO-8 font, which lives in the console, not the cart. The obvious shortcut was fake-08's fontdata.cpp. Checked rather than assumed:

  • fake-08 declares MIT (LICENSE.MD); GitHub reports NOASSERTION as it is an aggregate file. It credits ~15 components (z8lua, zepto8, tac08, PicoLove, SDL, LodePNG…) and says nothing about the font.
  • fontdata.cpp carries no copyright notice — its whole provenance is //taken from tac08. The chain is PICO-8 → tac08 → fake-08 → us, and MIT's one obligation (preserve the notice) has nothing to preserve. An emulator can license its own code; it cannot grant rights to data taken from PICO-8.

All moot — the font is CC-0 at the source. Lexaloffle's FAQ: “Yes, please do. The palette and font are both available under a CC-0 license.” (announcement). No attribution, no notice, no licence file — and the same sentence retroactively covers the PAL888 palette already in host_main.cpp.

Decision: take the glyph data from the source; do not vendor an emulator's font file. The data is CC-0 wherever it is read, but a specific file is that project's MIT work — transcribe the data, don't copy the file. This also avoids needing a Porting / adapting upstream code code-map doc: we are not porting anyone's code.

Trademark is separate and stricter. The same FAQ: “If you are making a separate software product, a run of multiple hardware units, PICO-8 related merchandise, or anything else -- please write to me first.” A personal DIY handheld is fine; more than one unit, or shipping under the PICO-8 name, is a write-first situation. Easy to misread “the font is CC-0” as “the branding is free” — the FAQ says the opposite in the same breath.

What's verified / what isn't

  • Verified: __gfx__ decode (by eye, from the generated header); the shared-region-is-map-data claim (4096 B, statistics matching __map__); the level→map-row mapping (read from the cart, cited); Gate #2's scoping (read from the bench, cited); font licensing (vendor's FAQ).
  • Not verified: nothing renders yet — spr is still a stub. The 16 KB unpacked-sheet decision is reasoned, not measured against a real spr() inner loop. Whether levels 16–30 move the Gate #2 average is unknown, not assumed.
