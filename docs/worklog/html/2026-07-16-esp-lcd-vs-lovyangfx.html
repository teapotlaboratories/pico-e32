Trying esp_lcd i80 instead of LovyanGFX: why colour “broke”, and a real comparison

⏹ CONCLUDED — esp_lcd REMOVED, LovyanGFX only
esp_lcd i80 is 1.5× faster (590 vs 393 fps, real zero-copy DMA) but its colour is broken on this board in a way byte-swapping does NOT fix — every bright fill renders the same teal, swap-invariant. Root-causing it needs a logic analyzer or owner board-wiring insight (DP-2). DP-1 resolved: the “colour wrong” code comments were overclaims — esp_lcd had never actually been tested on the correct pins before now. Follow-up (same day): the esp_lcd backend and the whole components/ili9488 wrapper were removed; the board now owns its display in board.cpp — see the Postscript.
Read this as a story with a twist. The middle builds a clean source-derived theory — “it's just a byte swap” — and then the hardware refutes it. The refuted theory is kept, not deleted: the reasoning was sound and the refutation is the finding.

The contradiction — in three places at once

The claim that esp_lcd “gets colour wrong” was asserted in the code three times and denied by the worklog once — all in main:

wheresays
ili9488.cpp:5-9esp_lcd was retried on rev-1 pins, “gets COLOUR wrong: red/yellow → blue/green, two hues not four”
CMakeLists.txt:3-5same — “gets colour wrong … so it is rejected”
panel-rev1-pinmap.md:86“esp_lcd i80 is untested on the correct pins”
…pinmap.md:115lists “Consider retrying esp_lcd” as a next step

Either it was retried and failed, or it was never retried. One is wrong. This session settled it on the bench.

The byte-swap theory — sound, and wrong REFUTED below

Formed from reading source before flashing. LovyanGFX byte-swaps every pixel in hardware via the GPIO matrix; esp_lcd does not. Bus_Parallel16.cpp:160-166 crosses the two bytes:

for (size_t i = 0; i < 8; ++i) {
    LGFX_GPIO_MATRIX_OUT(pins[i]  , idx_base + i+8);   // D0..D7  -> LCD_DATA_OUT8..15  (HIGH byte)
    LGFX_GPIO_MATRIX_OUT(pins[i+8], idx_base + i  );   // D8..D15 -> LCD_DATA_OUT0..7   (LOW byte)
}

esp_lcd_new_i80_bus maps data_gpio_nums[i] → LCD_DATA_OUT<i> straight. So identical pixel bytes reach the panel byte-swapped between the two drivers — which would turn the PICO-8 warm palette (red/orange/yellow all share the high-byte red bits) toward blue, exactly the reported symptom. Three independent fixes exist (esp_lcd's flags.swap_color_bytes, reordering data_gpio_nums, or the software swap in ili9488_rgb565), and the rule seemed to be “exactly one net swap.” Clean theory. The hardware disagreed.

This is the DP-3 finding made real: LovyanGFX's pushImage is the non-DMA path; esp_lcd is genuine zero-copy DMA. Clock is a wash — esp_lcd also lands 40.000 MHz exactly (160 ÷ 2 ÷ 2).

End-to-end (palette-expand + scale + blit): esp_lcd 256.5 fps vs LovyanGFX 210.6. On speed, esp_lcd is the better path — though Gate #1 already passes 13× over on LovyanGFX, so throughput isn't actually needed.

Result 2 — colour: the twist theory refuted

Every configuration renders colour wrong. Judged on the static L-pattern and solid R/G/B fills:

configresult on the glass
straight pins, app pre-swap, flags.swap_color_bytes=1shredded, all-teal
straight pins, flags.swap_color_bytes=1, MADCTL 0x98shredded shapes, teal
data-pin crossover (mimic LovyanGFX), app pre-swapworse — uniform teal, no shapes: crossing the pins scrambles the 8-bit init COMMANDS that share the bus
straight pins, native RGB565 (no swap)recognisable shapes, teal; solid R=G=B fill → same teal
straight pins, app pre-swap (swapped)identical to native — solid R=G=B → same teal

A byte-order bug cannot be swap-invariant: swapping the two bytes of 0xF800 vs 0x00F8 must change the hue if the panel decodes the bytes at all. It doesn't. So the clean source theory is wrong.

So the old “red/yellow → blue/green” is directionally right (colour is wrong on esp_lcd) but the mechanism is deeper than the byte-order story. The bit-level prediction was sound as a prediction; the hardware is swap-invariant, which no byte-swapping fixes.

Working theory, and the record corrected

Working theory (needs a logic analyzer). LovyanGFX drives this panel via a GPIO-matrix data-pin crossover it applies to data while still getting commands out — it packs commands with the crossover in mind. esp_lcd shares one fixed pin map between data and commands: cross the pins and the init commands scramble; leave them straight and the colour bits don't land. The board's actual DB-pin wiring isn't on the silkscreen, so every theory here is a guess about a bus I can't see. Resolving it needs a logic analyzer on D0-D15 + WR + DC, or the owner's knowledge of the rev-1 data-bus wiring — the same class of board fact that cracked the pin map.

DP-1 resolved, honestly. The worklog side was right: esp_lcd was never actually tested on the correct pins before this session. The code comments stating “retried … gets colour wrong” as a measured result were overclaiming — the failure is real, but it was inherited from wrong-pin runs and a source prediction until now. This session both confirms colour is broken and shows the stated reason (a byte swap) is wrong. Comments in ili9488.cpp and CMakeLists.txt corrected.

DP-2 decision: stay on LovyanGFX

esp_lcd's 1.5× speed and cleaner DMA are appealing, but it does not render correct colour on this board, the cause is not a config toggle, and Gate #1 already passes 13× over on LovyanGFX — so there is no pressure to switch and a real, unsolved blocker against it. Keep LovyanGFX as the default; keep the esp_lcd backend in-tree (ILI9488_BACKEND=esp_lcd, neither file deleted) as the substrate for a logic-analyzer session. Revisit if a future heavy cart needs the throughput, or the owner has board-wiring insight.

Roadmap note: LovyanGFX also covers the Phase-2 4" ST7701 RGB board (Bus_RGB + Panel_ST7701, with a bundled Makerfabs 4" config) and SPI panels — one library across the whole roadmap, which reinforces keeping it despite the 132 MB submodule.

Postscript — esp_lcd removed, and the ili9488 wrapper with it

Since LovyanGFX is the sole backend, the two-backend components/ili9488 wrapper (a config struct + build-time selector) earned its keep no more. Both were removed and the board took ownership of its own display:

  • Deleted: components/ili9488/ (both backends + header + CMake), boards/…/board_pins.h, and the ILI9488_BACKEND Makefile plumbing.
  • Added: boards/makerfabs-ili9488-r1/board.{h,cpp} — the same LovyanGFX bring-up with this board's pins/orientation/byte-order baked in, behind a board-agnostic board_lcd_* API. Apps compile ${BOARD_DIR}/board.cpp into main and link LovyanGFX directly; no shared display component.
  • Apps: display-test and host call board_lcd_* instead of ili9488_*.

Why the whole wrapper, not just the esp_lcd file: with one board and one driver, the config-struct indirection only spread board facts across two files. The board owning its display keeps the pin map, orientation and byte order in one place — and a future board (the Phase-2 4" ST7701 RGB panel) becomes a new boards/<board>/board.{h,cpp}, not an app edit. The esp_lcd findings are preserved above; the code is recoverable from git (e0a21cc) if the teal is root-caused.

Verified after the refactor (behaviour-preserving): display-test still reports 393.0 / 210.6 fps (byte-identical), palette bars render in distinct hues, and the host L-pattern is UPRIGHT with correct corners. All four apps build clean (display-test, host, bench-cam, native host). Board left on the LovyanGFX display-test build.
