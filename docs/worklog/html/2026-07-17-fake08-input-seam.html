Input seam

2026-07-17 · fake-08 gets a compile-time-switchable input driver — with a serial backend the bench can drive

fake-08's Host::scanInput() was a stub returning zero (pause menu never opened). It now sits behind a compile-time-selectable input seam: one C interface, exactly one backend compiled per build (stub | serial | touch | i2c), chosen by INPUT_BACKEND. Serial lands first because it is the only backend an automated bench can drive — a rig can't press a button, but it can write bytes to the console UART. Verified on hardware (2026-07-17): keys driven over /dev/ttyUSB1 registered on-device while the console kept logging, and a bench-camera frame confirmed the square moved centre → bottom-right.

The seam — and why serial first

scanInput() is the one place every input source flows through. Touch and buttons can't be actuated by an automated bench (the rig only observes the panel), so an input path the agent can drive over the wire is the only way to exercise the whole _update/btn/btnp chain end-to-end. Build the seam plus the serial backend first (fully verifiable), and touch / I²C drop in behind the same interface later.

The interface (components/input/input.h) is deliberately tiny — a backend only ever reports held; the pressed-this-frame edge is computed once, above the seam:

enum { INPUT_LEFT=1<<0, INPUT_RIGHT=1<<1, INPUT_UP=1<<2, INPUT_DOWN=1<<3,
       INPUT_O=1<<4, INPUT_X=1<<5, INPUT_PAUSE=1<<6 };   /* == fake-08 P8_KEY_* order */
esp_err_t   input_init(void);          /* once, at first scanInput */
uint8_t     input_poll(void);          /* the currently-HELD mask, once per frame */
const char *input_backend_name(void);  /* for the boot log */

ESP32Host::scanInput() calls input_poll() for the held mask and derives the KDown edge itself (held & ~prevHeld) — shared across every backend. The fake08 wrapper REQUIRES input.

Top: INPUT_BACKEND compiles exactly one backend into input_poll(); scanInput() derives the KDown edge and drives btn(). Bottom: the serial backend makes that same path hardware-in-the-loop — bytes from pyserial ride UART0 to btn() to the panel, and the bench camera closes the loop.

Same lever as CELESTE/MEASURE_FPS: components/input/CMakeLists.txt compiles input_${INPUT_BACKEND}.c, and the default (stub) installs no UART driver, so a plain build behaves exactly as before.

make build APP=pico-e32-fake08 BOARD=makerfabs-ili9488-r1 DEFS='-D INPUT_BACKEND=serial'

-D lands in CMakeCache.txt and persists — pass it each time, or make fullclean to change it.

The serial protocol

One byte per key over the console UART (115200, /dev/ttyUSB1 = the board's CP2104). A key is held ~HOLD_FRAMES (≈6) frames then auto-released, so a single byte taps and repeated bytes hold. Upper/lower case both accepted; unknown bytes ignored. Each received key is logged (input.serial: r), so the receive→map step is verifiable from the serial log alone, independent of the camera.

Verified on hardware (2026-07-17)

  • Receive → map: sent r/d/l/u/o/x over /dev/ttyUSB1; every key registered in the log (input.serial: <key>) while boot/console logs kept flowing.
  • Full chain on the panel: a FORCE_FLASH_CART dev flag (ignore any SD cart, run the flash cart) let the input-reactive test cart run. Drove RIGHT×22 + DOWN×22 via pyserial; a bench-camera frame confirmed the square moved from centre to bottom-right (frame counter 64 → 148). Full chain pyserial → UART0 → scanInput → btn() → panel, camera-verified.
Coexistence caveat — resolved. Reading UART0 RX while the console logs on the same UART0 TX coexists cleanly — keys registered while boot/console logs kept flowing. No fallback to USB-Serial-JTAG needed.
