pico-e32 — worklog

Running record of non-trivial investigations and bring-up sessions. Each entry is a docs/worklog/YYYY-MM-DD-<slug>.md with a companion HTML render linked from this index (see .ai/AGENTS.md → Worklogs).

  • 2026-07-17 fake-08 port: vendoring + the ESP32 platform — draw-only milestone The runtime becomes a port of fake-08. Forked to teapotlaboratories/fake-08 @ pico-e32, added a platform/esp32 Host binding + IDF component (source/ byte-identical), and a pico-e32-fake08 app. It boots and renders a flash .p8 cart upright on the panel via fake-08's own GameLoop and the shared z8lua VM — retiring the "nobody has run fake-08 on an ESP32" unknown. Build-integration fixes (gnu++17, lodepng disk) kept out of source/. A raw-capture "90° rotation" turned out to be the 90°-mounted bench camera, not the display — corrected (DP-8).
  • 2026-07-16 HG-6: pal, camera, clip — the drawing surface is complete The last of the graphics surface: pal() (colour remap — Celeste's flash), camera() (screen shake), clip(). All three apply at the single px() chokepoint; identity defaults keep every earlier test byte-identical. Verified numerically and on a real room (camera shift + a pal(i,7) death-flash). HG-1…HG-6 done — every draw call a real Celeste frame needs is now in place.
  • 2026-07-16 HG-5: print(), and a font that carries no one else's licence Implemented print() + a 3×5 font. The glyphs are authored in-tree (not copied or transcribed from any emulator's file — the most conservative licensing choice), with PICO-8 geometry so layouts line up. Celeste's real UI strings — OLD SITE, DEATHS:12, MATT THORSON, TIME 3:07 — all render legibly. No Gate 3 regression.
  • 2026-07-16 HG-4: map(), and the other half of Celeste's world Implemented map() — a real Celeste room renders. Resolved the shared-memory finding: __map__ is only rooms 0–15 of 32; the rest are aliased onto the sprite sheet. The generator now emits the full 128×64 map, and rooms 16–31 render as real levels, not noise. Gate 2 unaffected (its map access clamps to the first half); the layer filter is a verified strict subset.
  • 2026-07-16 HG-3: spr(), the first real cart drawing Implemented PICO-8's spr(n,x,y,[w,h],[flip_x,flip_y]) — the core of every cart. Verified camera-free via the host frame-dump → PNG harness against Celeste's real sheet: a 64-sprite grid renders recognisable sprites, a 2×2 block assembles a full Madeline, and an asymmetric “F” flips to exact framebuffer mirrors. Caught the vacuous-test trap — Celeste's sprite 0 is blank. No Gate 3 regression.
  • 2026-07-16 Trying esp_lcd i80 instead of LovyanGFX: why colour “broke”, and a real comparison Built an esp_lcd i80 backend to settle whether it really renders “colour wrong” on the rev-1 pins. It's 1.5× faster (590 vs 393 fps, real zero-copy DMA) — but its colour is broken in a way byte-swapping doesn't fix: every bright fill renders the same teal, swap-invariant. A clean source theory (“just a byte swap”) that the hardware refuted. The old “colour wrong” comments were overclaims — esp_lcd had never been tested on the correct pins. Decision: stay on LovyanGFX.
  • 2026-07-16 Fixing the Y-flip, and measuring Gate #1 for the first time honestly Both of the pin-map worklog's open items closed. The glass is mounted upside-down — a board fact, now .mirror_y = true, verified upright by camera on the first try. Gate #1 PASSES: 393.0 fps blit-only / 210.6 fps end-to-end against a 610.4 fps ceiling. The fps line had been printing unread all along, next to a stale 20 MHz ceiling that made it look impossible. The watchdog was firing inside the measurement window; the plan's “wrong” 130 fps was right all along.
  • 2026-07-16 The panel was never broken: this board is rev 1, and its LCD is on different pins Two days of “the panel is dead” resolved: this unit is Makerfabs' first revision, LCD on WR=35/DC=36/CS=37 — their newer N16R8 board moved it to 18/17/46 because octal PSRAM eats those pins. 288 fps, matching checksums and a clean init were all true and meaningless: the bus clocked into unconnected pins. Driver now LovyanGFX; Gate #1's fps is void.
  • 2026-07-15 ESP32Host graphics: cart assets, the harness, and the font question Started the PICO-8 graphics surface. Sprite sheet extracted and verified by eye; the font is CC-0 straight from Lexaloffle, so no emulator's copy needed. Two findings: half of Celeste's map hides in memory shared with the sprite sheet, and Gate #2's 15.8 ms/frame is honest but scoped to half the game.
  • 2026-07-15 Bench camera: the hardware-in-the-loop capture rig The rig Gate #1/#3 were blocked on is up (M5Stack Timer Camera F, OV3660). Getting there meant proving the ESP-EYE's OV2640 dead — no answer on any of 306 pin pairs, any address, either I2C driver — plus two traps: idf.py's 460800 baud vs the FTDI bridge, and an I2C driver conflict that boot-loops the board. Its conclusions were wrong (dead panel / bad focus) — see the 2026-07-16 entry.
  • 2026-07-14 Phase-0 Gate #3: trivial cart end-to-end (minimal ESP32Host) z8lua + ili9488 + a trivial cart in one frame loop: 161.5 fps. Rendering verified byte-identical host↔device via framebuffer checksum (camera-independent).
  • 2026-07-14 Phase-0 Gate #1: ILI9488 i80 scaled-blit FPS Display-test on hardware: 288 fps @ 256² (~10× the 30 fps gate, ~94% of the i80 bus). VOID — that 288 fps was DMA into unconnected pins (wrong board revision's pin map); the panel showed nothing.
  • 2026-07-14 Phase-0 Gate #2: z8lua throughput on real hardware First on-device bring-up. New z8lua integration builds+runs on the ESP32-S3. Throughput was ~5–8× slow — traced to one build flag (-fno-jump-tables); re-enabling -fjump-tables gave ~3× → Gate #2 passable. Also: why PicoPico's float switch doesn't help us.
  • 2026-07-14 z8lua: vendored copy → git submodule Forked z8lua to teapotlaboratories/z8lua, moved the two local patches onto branch pico-e32, and wired it as a submodule. Compiles .c as C++.
