What landed

  • spr(n,x,y,[w,h],[flip_x,flip_y]) — blits sprite n (8×8, or a w*8 × h*8 block) from the active sheet to (x,y), skipping transparent colours. Sprite n's top-left is (n%16*8, n/16*8); the flips mirror the whole block about its own centre.
  • g_gfx — the active sheet pointer (128×128 palette indices, HG-1's CELESTE_GFX layout). Null-safe: the trivial cart sets no sheet, so spr() is a no-op there and the Gate 3 output is untouched.
  • Minimal palt() — default colour-0 transparency + control. The colour remap half of pal/palt stays HG-6.

The gotcha: Celeste's sprite 0 is blank

The flip test first used sprite 0 — and passed vacuously. Sprite 0 is blank in Celeste (Madeline starts at sprite 1), so “flip of nothing == mirror of nothing” is trivially true and proves nothing. Fix: paint a known asymmetric, non-blank “F” into sprite 64 (below the grid) and flip-test that, plus assert the F is asymmetric so the mirror check can't pass for free. The L-pattern lesson again: an asymmetric known pattern beats “looks roughly right.”

No regression, and what's next

The trivial cart (Gate 3 / device path) still prints fb_hash=5e851a75 — byte-identical to before HG-3. spr() being null-safe on g_gfx means registering it changed nothing for a cart that never calls it.

  • HG-4 — map() is next, with the shared-memory gotcha: __map__ alone is only rooms 0–15 of 32; the generator must also emit sheet rows 64–127 as map rows 32–63 (and CELESTE_MAP[4096] is used by luabench — resize with care).
  • HG-6 adds the pal() colour remap; spr() honours the transparency table but not a remap yet.
  • The panel itself is a separate camera check (BC-2/BC-3); the host↔device fb_hash compare ties the host proof to the device.
