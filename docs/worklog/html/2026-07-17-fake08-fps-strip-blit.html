fake-08 on-device fps: a 4.6× drawFrame fix

2026-07-17 · measurement → optimization

Measured fake-08's real frame time on the ESP32-S3 (nobody had — Gate #2 measured the VM with drawing stubbed). Found drawFrame at 16.5 ms, ~9× the VM cost, because the 128 KB scaled framebuffer didn't fit internal SRAM and fell back to PSRAM. Fixed by blitting in strips from a 16 KB internal buffer (+ inlined nibble-unpack): 16.5 → 3.6 ms, frame ceiling 54 → 189 fps. The draw path is no longer the fps bottleneck.

How it was measured

An opt-in build (DEFS='-D CELESTE=1 -D MEASURE_FPS=1') swaps fake-08's GameLoop for an instrumented loop that times vm->Step() (the VM's _update/_draw) vs host->drawFrame() (our unpack → RGB565 → 2× → blit) separately with esp_timer, logging every 30 frames over UART (read with raw stty 115200 + cat). Target = the real Celeste cart; it parks on the title (input stubbed), but drawFrame's cost is content-independent so it's representative regardless.

The finding

The boot log gave the cause away: scaled fb fell back to PSRAM, fb=0x3c1004f8 (PSRAM lives at 0x3C000000+; internal DRAM at 0x3FC88000+). The 256×256 scaled frame is 128 KB — too big for internal DMA-capable SRAM — so it fell to PSRAM, and both the per-pixel writes and the blit-read then paid PSRAM latency.

The fix — strips from internal SRAM

Instead of one 128 KB framebuffer, build + blit the frame in 8 strips of 32 panel rows through a 16 KB internal DMA buffer (strip=0x3fcc99c8 — internal, no fallback). Every write and the blit DMA now stay in fast internal SRAM. Also inlined the 4-bpp nibble-unpack (one byte → two pixels), dropping the per-pixel getPixelNibble call. In the fork's ESP32Host::drawFrame (@8f4b6a6); source/ untouched.

What it means for frame rate

drawFrame is now 3.6 ms and content-independent. With Gate #2's measured gameplay VM cost (~15.8 ms avg, 5–40 ms per room):

  • avg room ≈ 15.8 + 3.6 = ~19 ms → ~51 fps — comfortable 30, near 60
  • light rooms ≈ 8 ms → 60 fps capable
  • densest rooms ≈ 44 ms → ~23 fps — now VM-bound, not draw-bound

So the draw path is settled; frame rate is now bounded by the z8lua interpreter on heavy rooms — to be tackled only after profiling a representative cart on the S3.
