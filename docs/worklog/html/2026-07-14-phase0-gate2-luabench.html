Phase-0 Gate #2: z8lua throughput on real hardware

✓ Gate #2 passable
The new z8lua build (compile .c as C++) runs on the ESP32-S3. First measurement was ~5–8× too slow — but the entire deficit traced to one ESP-IDF build flag, -fno-jump-tables. Re-enabling -fjump-tables gave ~2.8–3.1×, landing throughput ~2.5× under the PICO-8 target — inside the go/no-go window. No source edit.

Hardware & build

  • Board on /dev/ttyUSB0: ESP32-S3 rev1, 2 MB PSRAM, 16 MB flash — the Makerfabs ILI9488 N16R2. Vendored ESP-IDF v5.4.2, xtensa GNU 14.2.0.
  • New z8lua integration validated on target: the log shows 30 z8lua .c files compiled as CXX objects, 0 errors — set_source_files_properties(… LANGUAGE CXX) works on the real toolchain, no renames.
  • First flash tripped the Task WDT (the timed loop starves IDLE0); disabled it for the benchmark (CONFIG_ESP_TASK_WDT_INIT=n).

The hunt: what actually cost the cycles

Baseline fixed-point was ~420 cycles per VM instruction (~15× a normal Lua). I chased it lever by lever — most were red herrings:

lever triedeffectverdict
QIO flash + 32 KB I-cache0%✗ not fetch-bound
fix32 divide (isolated w/ no-divide case)5%✗ not divide
assertions—✗ lua_assert is a no-op
always_inline the fix32 operators1–6%✗ inlining wasn't it
IRAM placement—✗ predicted null (not fetch-bound)
fix32 double→float0%✗ math is fixed-point
-fjump-tables2.8–3.1×✓ the fix

Root cause: opcode dispatch was a compare-chain

With operators inlined the empty loop was still ~235 cyc/instruction → the cost was VM dispatch itself. ESP-IDF compiles with -fno-jump-tables -fno-tree-switch-conversion, so luaV_execute's ~40-way opcode switch becomes a linear compare-chain — every bytecode dispatch walks ~20 branches. That's invisible to cache, inlining, and divide, which is exactly why every other lever did nothing.

One build flag turns ~20 compares per bytecode into a single table jump.

Result & where it sits

casebaseline+jump-tablesspeedup
fixed-point arithmetic19281 ms6922 ms2.79×
arith, no divide18339 ms5921 ms3.10×
pico-8 sin/cos3435 ms1343 ms2.56×

Fixed-point 0.104 → 0.289 M iter/s ≈ ~1.6 M Lua VM inst/s vs the ~4 M/s PICO-8 target → ~2.5× off, inside the ~2–3× Gate #2 pass window. Separately, table-churn OOM'd on internal SRAM (369 KB) → the Lua heap needs PSRAM (reinforces the N16R8 / 8 MB production module).

Why PicoPico used float — and why we don't

Asked whether PicoPico's z8lua is a better base. Findings: our base is jtothebell/z8lua = fake-08's z8lua (our Phase-1 runtime), and it's more current (2026) than DavidVentura's fork (2023). PicoPico can't be adopted (no license; its ESP port drives ST7789/ILI9340 SPI, not our i80/RGB) — it's a reference, not a base.

Its double→float switch (May 2022, commits f907376/8b69c0c) was for the RP2040 — a chip with no FPU, where software float beats software double. On the S3, David's own LOG.md concludes the opposite — quoting Espressif, "even though ESP32 has a single-precision FPU, floating point is always slower than integer; use fixed point" — and measured sin fixed 135 ms vs float 735 ms (~7×). z8lua is already fixed-point, so float gave us 0 gain — reverted. (And PicoPico itself doesn't set -fjump-tables — our ~3× win is unexploited there.)
