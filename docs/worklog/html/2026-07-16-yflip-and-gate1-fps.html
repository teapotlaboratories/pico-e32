✓ Y-FLIP FIXED ✓ GATE #1 PASSES
Picks up from the pin-map worklog, which left exactly two things open. The glass on this board is mounted upside-down — a board fact, now recorded as .mirror_y = true and verified upright by camera on the first try. And the display path runs at 393.0 fps blit-only / 210.6 fps end-to-end against a 610.4 fps bus ceiling — 7–13× the gate.

A rig gotcha this session found: esptool.py flash_id — the very command the bench doc tells you to run to tell the two boards apart — power-cycles the target. The camera's tuned awb=0&exp=600&gain=0&sat=2 lives in the sensor's registers, not flash, so identifying the camera silently reverts it to auto-exposure and auto-white-balance. A bare /capture afterwards then quietly measures with exactly the AWB the rig exists to avoid. Now documented in the bench doc.

The watchdog was firing inside the measurement window

Not previously recorded. The Gate #1 loop never yielded, so IDLE0 starved for the whole 5 s run:

E (5782) task_wdt: Task watchdog got triggered. The following tasks/users did not reset the watchdog in time:
E (5782) task_wdt:  - IDLE0 (CPU 0)

It fires at t≈5.78 s — inside the 5.01 s window (which starts after init at t≈0.8 s). Printing a backtrace over a 115200 UART is not free and it lands in the timed region, so it could only have depressed 370.3 fps. Prediction: the true rate is that or higher. It was. With the yield added: 393.0 fps, +6.1%.

Flashed the L-pattern cart and captured. Rather than judge by eye — the failure mode this rig has already caused twice — the capture was blob-detected and mapped back into cart coordinates. The full-width L bar spans 128 cart px across 928 capture px → 7.25 capture px per cart px, which anchors everything.

The one assumption this rested on, stated honestly. All of it is conditional on the bench doc's claim that the camera is mounted 90° CW (LEFT of frame = TOP of panel). If that note were wrong by 180°, the identical capture would instead mean an X-flip. Nothing in the frame settles it — the rig cannot see its own mounting, and the 256×256 blit is centred, so even the panel borders offer no anchor. It didn't need settling analytically: the fix loop settles it. Apply a Y-flip → upright means the diagnosis was right; 180° means it was an X-flip. It came up upright, so the mount note is confirmed as a side effect.

The fix — offset_rotation = 4, and why it is the only candidate

LovyanGFX owns orientation, and its rotation arithmetic is not the obvious (r + offset) & 7. From Panel_LCD.cpp:130:

_internal_rotation = ((r + _cfg.offset_rotation) & 3) | ((r & 4) ^ (_cfg.offset_rotation & 4));

Bit 2 is a mirror flag XORed in — it can never carry into the rotation bits. With the app's rotation at 0, offset_rotation = 4 → _internal_rotation = 4. The MADCTL table lives at Panel_LCD.hpp:85-99; Panel_ILI9488 does not override getMadCtl — the override at Panel_ILI948x.hpp:87 belongs to the sibling Panel_ILI9481, which is easy to misread.

_internal_rotationMADCTL (+BGR)MV (axis swap)MXMY
00x08000← was
20xDC011flips X too

1, 3, 5, 7—1——all swap to 480×320

The bit semantics are documented in-tree, not folklore — vendor/esp-idf/components/esp_lcd/include/esp_lcd_panel_commands.h:51-56: MY (1<<7) “Row address order, 0: top to bottom, 1: bottom to top” — the mirror; ML (1<<4) “Line address order … refresh top to bottom / bottom to top”. So rotation 4's MY|ML is a vertical mirror with the panel's refresh direction brought along to match, which is why LovyanGFX pairs them in every entry: scanout travels the same way as addressing, avoiding a tear. (That header carries its own caveat at :48-50 — these positions are conventional, not standardised. It corroborates LovyanGFX's table; the hardware result is what settles it.)

4 is the only value in 0–7 that inverts Y while leaving MX and MV clear. Geometry is provably untouched: 4 & 1 == 0 so no dimension swap (Panel_LCD.cpp:138); 4 & 2 == 0 so _colstart = 0 (:146); and _rowstart = mh - (ph + oy) = 480 - 480 = 0 (:149) — which lands on zero only because memory_height == panel_height and offset_y == 0, both true here. Cost: one MADCTL byte at init, zero per frame.

Where it lives matters more than the value

In boards/makerfabs-ili9488-r1/board_pins.h, as .mirror_y = true, replacing the dead .madctl = 0x48 — not in the driver. The glass on this board is mounted upside-down relative to the controller's native scan direction: a fact about this board, not about the ILI9488 and not about LovyanGFX. It belongs in the file that calls itself “THE ONE DEFINITION” for board facts. The driver just asks c->mirror_y ? 4 : 0, so ili9488_config_t stays free of LovyanGFX's rotation encoding — which matters, because retrying esp_lcd is still an open option and mirror_y survives that swap where an offset_rotation byte would not.

The vendor never ran rotation 0. The rev-1 source of record (LGFX_MakerFabs_Parallel_S3.hpp:106) also sets offset_rotation = 0 — our config was byte-identical to the vendor's. But their demo calls lcd.setRotation(1) (main.cpp:134, landscape 480×320). _internal_rotation = 0 on this board was untested territory upstream, so the reference matching us was never evidence that rotation 0 renders correctly. It is the pin-map trap one layer in: a vendor's file is only authoritative for the configuration the vendor actually ran.

The dead .madctl = 0x48 named MX|BGR; the fix needs MY. It disagreed — and carries no weight. It was never applied by any driver (0x08 was what actually went to the panel), and it was set during the era when the bus clocked into unconnected pins, so it was never validated against a lit panel either. Not a second opinion; noise that looked like config.

Both reported numbers clear the gate by 7–13×. The display path is not the bottleneck — as the plan predicted.

I (802)  trackA: bus ceiling 610.4 fps  (16-bit @ 40.0 MHz, 256x256 RGB565, 1 cycle/px)
I (2422) trackA: blit-only   600 frames in 1526.638 ms ->  393.0 fps  (frame 2.54 ms mean, 2.54 min, 2.55 max)  64.4% of ceiling
I (4002) trackA: end-to-end  300 frames in 1424.641 ms ->  210.6 fps  (frame 4.75 ms mean, 4.75 min, 4.75 max)  34.5% of ceiling

Spread is essentially zero — 2.54 min / 2.55 max over 600 frames — and eight consecutive passes reproduce to 0.1 fps. A genuinely sustained rate, not an average hiding stalls. The expand+scale costs 4.75 − 2.54 = 2.21 ms/frame: the CPU-side palette work is comparable to the entire bus transfer.

The ceiling, re-derived — and the old one was never wrong

The ILI9488's notorious RGB666/18-bit coercion (2 writes/pixel) is SPI-only: LovyanGFX applies it only when busType() == bus_spi (Panel_ILI948x.hpp:169-179). On Bus_Parallel16 the panel gets COLMOD 0x55 = RGB565, and dlen_16bit puts 16 bits on the wire per WR strobe → one bus cycle per pixel. 40 MHz is exact, not nominal: 240 MHz ÷ 2 ÷ 3, no fractional divider.

A correction to the record, in the project's favour. The Gate #1 worklog declared the plan's ≈130 fps theoretical @16-bit/20MHz “wrong” and replaced it with ~305. The 130 was right. It is a full-panel 320×480 figure: 153,600 ÷ 20e6 = 130.2 ✓, and its companion “≈32 fps at 8-bit/10 MHz” checks out too (307,200 ÷ 10e6 = 32.6 ✓). The worklog compared a 320×480 ceiling against a 256×256 measurement, called the ceiling wrong, and overwrote it — which is how the harness came to print a stale, mismatched “~305 fps @16-bit/20MHz”. A correct number was destroyed by comparing it against a different quantity. The harness now derives the ceiling from cfg.pclk_hz and logs an error if a measurement ever exceeds it.

Why this isn't another 288

393 fps proves the ESP32's LCD_CAM finished strobing WR. It proves nothing about the panel — bus timing is identical whether the pins reach an ILI9488 or nothing at all. That gap is unclosable by timing, and it is exactly the gap the 288 fps fell through. Only the camera closes it.

Verified in source: Panel_LCD.cpp:87 → Bus_Parallel16.cpp:211-216. The loop measures completed transfers.

The camera evidence: the panel is live during the timed window

Four captures, 1 s apart, while the measurement loop ran:

  • Content changes between every pair. Mean |difference| across the full frame: 12.1 – 24.3 levels. The panel region's mean RGB swings R 31.6→45.5, G 56.9→91.1. Far above sensor noise.
  • The two phases are visually distinguishable, and they alternate. live-1 and live-3 show 16 clean stripes — captured during blit-only, where the content is static. live-2 and live-4 show a torn patchwork — captured during end-to-end, where the palette advances every frame. The tear bands run along panel rows (the 90° mount maps panel-y to frame-x): GRAM is rewritten at 210 fps while the panel scans its glass at ~60 Hz (Panel_ILI948x.hpp:209: FRMCTR1 = 0xA0), so different rows hold different palette generations at the instant of exposure. 2 clean / 2 torn, alternating, matches the 1.53 s / 1.42 s phase split.
Honesty note — a measurement I attempted and threw away. I tried to count tear bands programmatically to quantify the write rate. The detector is unreliable and its numbers are not reported: at this focus the LCD pixel grid is resolvable (the rig working as designed), and that high-frequency pattern registers as “steps” along the same axis as a tear. Smoothing above the grid pitch then picked up the panel edge instead, and it flagged live-3 — plainly clean to the eye — as torn. The tearing is real and visible; my count of it was not trustworthy, so it stays qualitative. An invented band-count would have been this session's “double byte-swap”.

What is still NOT proven, and cannot be by anything above: that every pixel latches cleanly. The panel is clocked at 40 MHz (25 ns WR cycle) against a datasheet 8080-II minimum of 50 ns (20 MHz) — 2× its rating. That value comes from the vendor's own rev-1 config (LGFX_MakerFabs_Parallel_S3.hpp:76) and it renders, which is the only evidence behind it; there is no timing-margin analysis. Marginal latching would show as sparkle or dropped pixels, not as a wrong fps. Judge from the frame, never the counter.

One thing the number must never be read as. 393 fps is a GRAM-write rate, not photons per second. The panel scans its own glass at ~60 Hz, so nothing above ~60 is ever visible — to the eye or the camera. The tearing above is that fact made visible. This doesn't weaken Gate #1, which asks whether the blit path can sustain 30 fps of work (it can, 13× over), but “the screen updates 393 times a second” would be false.

Found on the way — filed, not fixed

All in the new display area doc as DP-1…DP-7. Deliberately not bundled in: Gate #1 passes 13× over without any of them, and the Y-flip fix should be reviewable on its own.

  • DP-1 — the repo contradicts itself about esp_lcd, in main, right now. ili9488.cpp:5-9 says esp_lcd i80 was retried on the correct rev-1 pins and “gets COLOUR wrong: red and yellow come out blue/green, only two distinguishable hues instead of four”. The pin-map worklog says it was never tried (:86) and lists “consider retrying” as a next step (:115). Committed 36 seconds apart (66b3958, 080d68f). One is false and nothing marks which. Given this rig's record, the colour claim is the one to doubt first — but it was never written up, so it is flagged, not overruled.
  • DP-3 — blits never use DMA. use_dma defaults to false. Explains most of the 393→610 gap. Has a real trap: DMA reads the caller's buffer, so it and the harness's rebuild-in-place are individually safe and jointly a tearing race.
  • DP-4 — the checksum's coverage is oversold. It hashes the indexed framebuffer only; everything from palette expansion to the wire — exactly where the two-day bug lived — is outside it. Setting .swap_color_bytes = false would double-swap the panel's colours while the hash stays identical.
  • DP-5 — the negative control. Misconfigure WR/CS, confirm fps is unchanged while the camera goes blank. Gate #1's own worklog predicted the 288 fps failure in writing and it was written off as hypothetical. Running it once makes it a fact.
  • DP-6 ili9488_fill is slow, redundant and has a latent OOB. DP-7 bus_shared = true is inert but its comment is false.
