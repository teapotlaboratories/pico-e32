The panel was never broken: this board is rev 1, and its LCD is on different pins

✓ RESOLVED — the display works
After ~two days of concluding the panel was dead, the cause was the pin map. This unit is Makerfabs' first revision, whose LCD sits on WR=35 / DC=36 / CS=37 — not the 18 / 17 / 46 their current board and current example code use. Every command we sent went to pins this board does not wire to the panel.
The one-line lesson: “verified against the vendor's own source” is worthless if you never check which revision that source describes.

Why two maps exist — it is the PSRAM

Octal PSRAM consumes GPIO 35/36/37, so the newer board had to move its LCD. Data bus D0–D15 is identical on both — only WR/DC/CS moved, which is why nothing looked obviously wrong.

What the failure looked like

A backlit white panel, forever — while everything that could report success, did: the i80 bus built, DMA completed, 288 fps sustained, framebuffer checksums byte-identical host↔device, panel init ESP_OK, the component logging “Initialization complete”. All true; all meaningless — the bus was clocking into pins with nothing on the other end.

Gate #1's own worklog predicted this in writing: “a broken panel init could still clock DMA at 288 fps into a blank screen.” It was right, and it still took two days.

The two facts that broke it open were the owner's, not measurements: “the LCD works from the factory, only after you flash it does it stop” (kills dead-hardware) and “this is the first revision” (the actual answer). The decisive move was theirs too — pointing at radiosound-com/makerfabs-parallel-tft-lvgl-lgfx, whose S3 header shows WR 35, RS 36, CS 37.

Makerfabs' own example failing was the most misleading evidence of all. It built and booted cleanly and still showed nothing — reading as “even the vendor's code can't drive it, so the hardware is dead.” It is written for their newer board. A vendor example is authoritative only for the revision it targets.

Driver: now LovyanGFX — the plan's other branch

components/ili9488 now delegates to LovyanGFX (submodule pinned 1.2.25, FreeBSD licence) with the factory sketch's rev-1 config: Bus_Parallel16 + Panel_ILI9488, dlen_16bit=true, pin_cs=37, 40 MHz. Public API unchanged; both apps compile untouched.

The plan said so all along — §A1: “start from atanisoft/esp_lcd_ili9488 or a LovyanGFX Bus_Parallel16 config”. The original Gate #1 driver followed neither (hand-rolled); the first branch was then forced repeatedly after it failed. esp_lcd i80 remains untested on the correct pins.

What the camera proved, in the end

The rig was right the whole time and was disbelieved. It reported a uniform blue-white field that never responded to LCD content — exactly what a backlit, unaddressed panel looks like. Two theories were built to explain it away rather than doubting the pin map.

  • ✅ 16 vertical palette bars, 256×256 centred — 57 px/bar, LCD pixel grid visible
  • ⚠️ Panel renders Y-flipped — caught by an asymmetric L-shape, so it is camera-artifact-proof. Open (MADCTL / LovyanGFX mirror).
  • ⏸️ Colour: not assessable with this rig. A confidently-diagnosed “double byte-swap” was invented from camera colour casts and withdrawn.

Now unknown again

Gate #1's 288 fps is void — DMA into unconnected pins, at 20 MHz on esp_lcd. The bus is now LovyanGFX at 40 MHz actually driving a panel. That measurement has never been taken.
