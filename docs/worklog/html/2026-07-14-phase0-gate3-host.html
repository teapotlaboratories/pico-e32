Phase-0 Gate #3: trivial cart end-to-end (minimal ESP32Host)

✓ groundwork passed ⚠ panel image unverified
A trivial PICO-8 cart runs end-to-end on z8lua + components/ili9488 at 161.5 fps (~5.4× the 30 fps gate). Rendering verified byte-identical host↔device via a framebuffer checksum — the right pixels reach the framebuffer without needing the camera.

The pipeline (firmware/pico-e32-host)

The minimal ESP32Host: everything from cart Lua to panel pixels, 161.5 fps.

Verification: framebuffer checksum (no camera)

fb_hash = FNV-1a over the 128×128 framebuffer, per frame. The cart + z8lua fix32 + draw API are deterministic, so host and device must produce the same sequence if the pipeline is correct:

host   0-9: 432b2a8d b99c6cb6 a3304eb4 58e93d00 2d861cfc afd99258 a8a681d0 a5fb37c3 01d10580 2e41c322
device 0-9: 432b2a8d b99c6cb6 a3304eb4 58e93d00 2d861cfc afd99258 a8a681d0 a5fb37c3 01d10580 2e41c322   ✓ identical

Byte-for-byte match → correct pixels reach the framebuffer on hardware; hashes change each frame → the cart animates. (The host build also caught a Lua-stack bug under ASan before any flash.)

Result

  • 161.5 fps (`_update`+`_draw`+scale+256² blit) — ~5.4× the 30 fps gate, ~2.7× 60 fps. Raw blit alone was 288 fps (Gate #1); cart draw + scale add ~2.7 ms/frame.
  • App 378 KB, 64% free; frame loop yields on the blit's DMA wait (no WDT issue).
  • ⚠ Not verified: the physical panel image (colours/orientation on glass) — same camera blocker as Gate #1.
